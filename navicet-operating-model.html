Phase 1 Define Ideation → PRD
Initiate & lock
Initiates project idea. Conducts structured intake interview with Claude Chat. Reviews, challenges, and edits Idea Brief, PRD, and Design Doc drafts. Locks documents at v1.0.
Handoff checklist
Confirms PRD locked, Design Doc locked, wireframes present, architecture diagrams present, tech stack confirmed, risks reviewed.
Claude Chat
Runs intake interview. Generates Idea Brief. Drafts PRD (all sections incl. REQ-IDs, AC layers, risks). Drafts Design Doc (UX flows, component inventory, data model, API contracts). Flags ambiguities for the Human to resolve.
Gate 1
◆
PRD + Design Doc locked at v1.0 · All handoff checklist items confirmed Owner: Human · Blocker: FORGE cannot start until this gate passes
Phase 2 Design UX + Technical
Review & approve
Reviews wireframes and architecture diagrams. Approves or requests changes. Signs off on tech stack and data model. No design decisions made unilaterally by AI.
Atlas (orchestrator) — session start protocol
Reads PRD + Design Doc. Summarises: core function, all P0 REQ-IDs, tech stack, open risks. Awaits Human acknowledgement before any build begins.
FORGE — Iris (UX) + Hector (Frontend)
Produce wireframes (WF-SCR-01 through SCR-0n) in /wireframes/. Annotate with CMP-IDs. Default, loading, empty, and error states for each screen.
FORGE — Cleo (Backend) + Hermes (Infra)
Produce architecture diagram (/diagrams/). Finalise OpenAPI spec. Define data model. Confirm tech stack rationale. Surface any technical risks.
Gate 2
◆
Human acknowledges session start summary · Wireframes and architecture diagram reviewed and approved Owner: Human · Blocker: no build work begins until explicit acknowledgement
Phase 3 — Execute
Phase 3a Build P0 requirements
Ad hoc check-ins
Available for scope questions, ambiguity escalations, and P1 confirmation requests. Does not unilaterally change PRD or Design Doc. Logs any needed deviations.
Deviation approval
Reviews DEVIATIONS.md entries raised by FORGE. Approves or rejects scope deviations before they are built. Resolves ambiguous requirements on request.
FORGE — Atlas (orchestrator)
Manages build sequence against locked PRD. Assigns work to sub-agents by REQ-ID. Surfaces scope deviations and ambiguities to the Human. Never expands scope silently.
Hector · Cleo · Hermes (sub-agents)
Build against P0 requirements only. Hector: UI components and screens. Cleo: API server, business logic, data layer. Hermes: integrations, OAuth flows, environment config.
Iris (sub-agent) — ongoing UX oversight
Available to Hector for UX decisions during build. No unilateral design changes — surfaces to Human if implementation diverges from wireframes.
Gate 3
◇
P1 feature confirmation (if time and complexity permit) Owner: Human · Conditional: FORGE pauses on P1 items until confirmed; can fail with DEVIATIONS.md entry
Phase 3b QA Acceptance criteria
UAT & spot checks
Reviews Argus QA report. Performs user acceptance testing on end-to-end flows. Flags any failures or gaps. Final sign-off authority on QA pass.
Argus (QA sub-agent)
Validates every P0 acceptance criterion across all 4 layers: plain English, Given/When/Then, input→output, edge cases. Signs off each feature before it is considered done. Produces QA report per REQ-ID. Flags failures to Atlas.
Atlas — QA orchestration
Tracks QA status per REQ-ID. Escalates failures to the relevant sub-agent for fix. Does not mark a feature complete without Argus sign-off.
Gate 4
◆
All P0 ACs pass Argus sign-off · Human UAT complete · No open P0 failures Owner: Human (final) + Argus (AI) · Blocker: no release until both have signed off
Gate 5 — Architecture Review
Gate 5 Architecture Review AI + Human
Architecture review (mandatory)
Reviews AI-generated architecture assessment. Validates security decisions, data handling, and constraint-specific risks. Checks scale assumptions. Makes final call on flagged risks. Documents decisions.
Risk sign-off
Accepts, mitigates, or escalates each risk flagged by FORGE. No risk is silently ignored. Architecture approved in writing before release.
FORGE — Hermes + Cleo (AI architecture review)
Produces architecture review document: checks built system against Design Doc technical sections, API contracts, data model, and scale constraints. Lists any deviations from spec.
Argus — security and compliance check
Reviews never-do list adherence, data isolation, and constraint compliance per PRD Section 6. Flags any violations for Human review.
Gate 5
◆
Architecture review complete · All risks documented and accepted by Human · Never-do list verified Owner: Human · AI review alone does not satisfy this gate — human sign-off is required
Phase 3c Release Deploy + go-live
Release decision
Authorises deployment. Reviews end-to-end simulation results. Confirms release scope (all P0, any confirmed P1). Approves go-live.
Hermes (sub-agent) — deployment
Executes deployment to agreed environment. Verifies integrations live. Runs end-to-end simulation. Produces release checklist for Human review.
Atlas — release documentation
Finalises DEVIATIONS.md. Confirms all deviations documented with REQ-ID, reason, and resolution. Archives project files in correct folder structure.
Gate 6
◆
End-to-end simulation passed · DEVIATIONS.md complete · Human authorises go-live Owner: Human · Blocker: Hermes simulation results reviewed before authorisation
Phase 4 Learn Retro (async, 5 days)
Retro review & sign-off
Reviews FORGE-drafted retro. Adds qualitative observations, product decisions, and strategic learnings. Approves and finalises RETRO_[project-name].md.
Atlas — retro draft
Drafts RETRO_[project-name].md: what was built vs scoped, all deviations with outcomes, QA findings, risks that materialised, and recommended process improvements for the next project.
Gate 7
◇
RETRO_[project-name].md finalised within 5 business days Owner: Human · Conditional: project lifecycle closes when retro is approved; can proceed if deferred with documented reason
Ad hoc human touchpoints (Phases 3a – 3c)
FORGE escalates to the Human when: (1) a requirement is ambiguous — do not interpret and proceed; (2) a scope deviation is needed — log in DEVIATIONS.md before building; (3) a P1 item is being considered; (4) a technical risk materialises that was not in PRD Section 8. The Human does not need to be available continuously — FORGE works autonomously and queues escalations.
